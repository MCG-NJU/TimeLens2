# 🎯 TimeLens2 Rollout and GRPO

## 🛠️ Install

```bash
pip install -r requirements.txt
```

## 🗂️ Data config and annotation format

The official recipe uses **both** training sources (TimeLens2-93K and TimeLens-100K). They are defined together in
`configs/data/timelens2.json`.

Each source provides its annotation path, media root, and prompt template:

```json
{
  "timelens2-93k": {
    "anno_path": "data/annotations/timelens2-93k.jsonl",
    "mp4_root": "/path/to/videos",
    "frames_root": "/path/to/extracted_frames",
    "prompt_template": "... {query} ..."
  },
  "timelens-100k": {
    "anno_path": "data/annotations/timelens-100k.jsonl",
    "mp4_root": "/path/to/videos",
    "prompt_template": "... {query} ..."
  }
}
```

The two annotation JSONL files are included under `data/annotations/`; only the
videos need to be downloaded. The checked-in media roots reproduce our
cluster layout. If the videos are stored elsewhere, edit `mp4_root` and
`frames_root` directly. A relative
`video_path` ending in `.mp4` is resolved under `mp4_root`; another relative
`video_path` is treated as a pre-extracted frame directory and resolved under
`frames_root`. Absolute media paths are used as-is.

Annotation JSONL supports either grouped or flat rows:

```json
{"video_path":"clip.mp4","duration":120.0,"events":[{"query":"...","span":[[10.0,15.0]]}]}
{"video_path":"clip.mp4","duration":120.0,"query":"...","span":[[10.0,15.0]]}
```

The grouped form is used as rollout input. Rollout emits the flat form with a
`rollouts` list and writes a new combined `train_data_config.json` for GRPO.

## 🎲 Rollout

`configs/data/timelens2.json` already points to both bundled annotations. Use
the SFT checkpoint produced by `../sft`; this one command rolls out both
`timelens2-93k` and `timelens-100k` from the combined config:

```bash
MODEL_PATH=/path/to/sft-4b bash scripts/rollout_timelens2.sh
```

The rollout script generates one worker per visible GPU and finally writes one
merged JSONL per source plus a combined `train_data_config.json` under
`outputs/rollout/timelens2/`. Use a fresh
`PRED_ROOT` for each model/data/seed combination: concurrent jobs writing the
same shard names will corrupt or overwrite one another. Do not start GRPO until
the merge succeeds and every training row has the expected 8 rollout responses.

## 🏋️ GRPO

The 4B recipe runs for 200
optimizer steps and the 8B recipe for 300.

The official rollout JSONL files are included under `data/rollouts/`. They were
generated by the 4B SFT checkpoint and are the rollout inputs used by the
original 4B and 8B GRPO recipes. Both launchers therefore work without rerunning
the complete rollout:

```bash
MODEL_PATH=/path/to/sft-4b bash scripts/train_grpo_4b.sh

MODEL_PATH=/path/to/sft-8b bash scripts/train_grpo_8b.sh
```

Both launchers default to `configs/data/timelens2_rollout.json`, which contains
the bundled rollout files for both `timelens2-93k` and `timelens-100k`. After
downloading the videos, edit only the media roots in that config.

To train from a newly generated rollout instead, override the default:

```bash
MODEL_PATH=/path/to/sft-4b \
DATA_CONFIG=outputs/rollout/timelens2/train_data_config.json \
bash scripts/train_grpo_4b.sh
```

Outputs are written under `outputs/grpo/timelens2-4b/` or
`outputs/grpo/timelens2-8b/`.
